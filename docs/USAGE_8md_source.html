37 ## Metrics
38 MetroCollect gathers metrics from different sources:
39  * CPU metrics from `/proc/stat`
40  * Memory metrics from `/proc/meminfo`
41  * Network metrics from `/proc/net/dev`
42  * Addtional network metrics with Ethtool
43 
44 Those metrics are organized hierarchically. For instance, `/cfm/memory/cached` represents the amount of cached memory on the system, while `cfm/network/eth0/rx/bytes` represents the number of bytes received by `eth0`.
45 
46 Most metric have a unit attached to it, and are expressed in units per second.
47 
48 ### CPU Metrics
49 CPU metrics are read from the `/proc/stat` file, you may refer to the relevant man page of your system to have a thorough description.
50 
51 As a summary, there are three types of CPU metrics:
52  - for each CPU core, proportion of time spent in differents CPU states (user, system, nice, idle, etc.)
53  - aggregated proportion of time spent in differents CPU states
54  - other CPU metrics: number of interrupts, soft interrupts, spawned processed, running and blocked processes, etc.
55 
56 ### Memory metrics
57 Memory metrics are read from the `/proc/memory` file, you may refer to the relevant man page of your system to have a thorough description.
58 
59 ### Network metrics
60 Network metrics are read from the `/proc/net/dev` file, you may refer to the relevant man page of your system to have a thorough description.
61 
62 For each interface (either up or down), the number of bytes, packets and various errors are collected, for both reception ("rx") and transmission ("tx").
63 
64 ### Ethtool metrics
65 These are the same metrics provided by the `ethtool -S $iface` for each network interface `$iface`. It provides very detailled information about each interface, but these information depend on the interface's driver. you may refer to the driver documentation for more details.
66 
67 MetroCollect supports all drivers supported by `ethtool`, and it is optimized for `ixgbe`, `igb` and `i40e`.
68 
69 When a number appears in a metric name, it is moved at the end of it. For instance:
70 ``` bash
71 $ ethtool -S eth0
72 rx_queue_0_packets: 0
73 rx_queue_0_bytes: 0
74 rx_queue_1_packets: 0
75 rx_queue_1_bytes: 0
76 ...
77 ```
78 becomes
79 ```
80 /ethtool/ixgbe/eth0/rx_queue_i_packets/0 0
81 /ethtool/ixgbe/eth0/rx_queue_i_bytes/0 0
82 /ethtool/ixgbe/eth0/rx_queue_i_packets/1 0
83 /ethtool/ixgbe/eth0/rx_queue_i_bytes/1 0
84 ...
85 ```

87 ## Standalone interfaces
88 ### MetroCollectValues
89 This program collects all available metrics and prints them on the standard output. For each metric, it shows values read from the kernel at some time, values read some time later, and the computed variation. That is because most kernel values are counters that can only increase; for example the number of bytes sent by eth0 or the amount of interrupts serviced.
90 
91 MetroCollectValues can take up to **two arguments**:
92 - the sampling interval in milliseconds, that is the duration to wait between two readings of kernel values. Default is 100 milliseconds
93 - how many times to report metrics. Default is 0, which means unlimited time
94 
95 For example, `./MetroCollectValues 500 10` will read kernel values and print them every 500 milliseconds, and it will do so 10 times. The program will thus run for 5 seconds.
96 
97 ### MetroCollectStats
98 This program collects all available metrics, compute statistics on a moving window and prints them on the standard output. For each metric, it shows the minimum, maximum, average and standard deviation of its variation.
99 
100 MetroCollectValues can take up to **four arguments**:
101 - the sampling interval in milliseconds, that is the duration to wait between two readings of kernel values. Default is 100 milliseconds
102 - the length of the moving window in number of samples. Default is 10 (statisctics are computed every 10 samples)
103 - the overlap of the moving window in number of samples, it must be less than the length of the window. Default is 0 (no overlap)
104 - how many times to report metrics. Default is 0, which means unlimited time
105 
106 By default, statsics are thus computed and printed every 1 second.
107 
108 For example, `./MetroCollectStats 50 5 0 4` will read kernel values every 50 milliseconds, and compute and print stats every 5 samples (that is every 250 milliseconds), and it will do so 4 times. The program will thus run for 1 seconds.
109 
110 
111 ### MetroCollectFiles
112 This program collects all available metrics and write them to a file.
113 
114 MetroCollectValues can take up to **three arguments**:
115 - the sampling interval in milliseconds, that is the duration to wait between two readings of kernel values. Default is 100 milliseconds
116 - how many times to report metrics. Default is 0, which means unlimited time
117 - path of the file to write to. Default is `./output.csv`
118 
119 For example, `./MetroCollectFile 0 20 maxspeed.csv` will read kernel values continuously (no delay between readings) 20 times, and write the variation to he file `./maxspeed.csv`.

122 ## Snap Configuration
123 ### Configuration
124 First set up the [Snap framework](https://github.com/intelsdi-x/snap/blob/master/README.md#getting-started).
125 
126 The default configuration for a MetroCollect Snap task is the following:
127 ```yaml
128 ---
129  version: 1
130  schedule:
131  type: "streaming"
132  max-failures: 10
133  workflow:
134  collect:
135  metrics:
136  /cfm/*: {}
137  config:
138  /cfm:
139  SendValues: false
140  SendStats: true
141  SamplingInterval: 100
142  ProcessingWindowLength: 10
143  ProcessingWindowOverlap: 0
144  ConvertToUnitPerSecond: true
145  UnchangedMetricTimeout: 120
146  # MaxMetricsBuffer: 0
147  # MaxCollectDuration: 0
148  publish:
149 # ...
150 ```
151 
152 ### Parameters
153 The parameters are (default values are given [above](#configuration)):
154  - `SendValues` (type boolean): whether or not to send values to Snap
155  - `SendStats` (type boolean): whether or not to send statistics to Snap
156  - `SamplingInterval` (type int): delay in millisecond between two readings of the kernel values
157  - `ProcessingWindowLength` (type int): moving window length (in number of samples). It is used to compute statistics
158  - `ProcessingWindowOverlap` (type int): moving window overlap (in number of samples), it must be less than the length of the window. It is used to compute statistics
159  - `ConvertToUnitPerSecond` (type boolean): wether or not to convert values in units per second if relevant (if you prefer the number of bytes sent per second instead of bytes sent per 100 milliseconds)
160  - `UnchangedMetricTimeout` (type int): by default, for performance reasons, constant metrics are downsampled and thus not sent everytime to Snap. If the value changes it is sent immediatly. This parameter is the downsample factor: how many null variations to ignore before sending the value to Snap again. If it is 0, downsample will not take place, if it is -1 constant values will never be sent
161  - `MaxMetricsBuffer` (type int): maximum number of metrics to send to Snap at once
162  - `MaxCollectDuration` (type int): maximum waiting time (in seconds) before sending metrics to Snap
163 
164 By default, values will be read from the kernel every 100 milliseconds, compute stats every 10 samples (that is every 1 second) and send them to Snap. If a value has remained constant across 120 iterations (that is during 2 minutes), zeros will be sent to Snap.

167 ### Metrics
168 Collected metrics are described [above](#metrics).
169 
170 For each metric, you can select which statistics you are interested in (shown below for the metric of aggegated CPU proportion of time spent in user mode, **note the use of wildcards**):
171 ```
172 /cfm/cpu/all/user/min/*: {}
173 /cfm/cpu/all/user/max/*: {}
174 /cfm/cpu/all/user/average/*: {}
175 /cfm/cpu/all/user/stddev/*: {}
176 ```
177 
178 You can use wildcards to specify groups of metrics easily:
179 ```
180 /cfm/cpu/all/*: {}
181 /cfm/memory/*: {}
182 /cfm/network/eth0/*: {}
183 ...
184 ```
